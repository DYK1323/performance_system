# 범용 성과관리 플랫폼 데이터 관리 정책서 초안

## 개요

이 문서는 범용 성과관리 플랫폼의 데이터 생성, 수정, 확정, 보존, 조회, 이력 관리 원칙을 정리한 초안이다.

- 목적: 기능 기획을 넘어 실제 운영 가능한 데이터 정책 기준을 마련
- 범위: `MetricLibraryItem`, `DataSource`, `StatusSet`, `Program`, `ProgramVersion`, `ProgramVersionItem`, 계획/실적/판정/증빙/권한/업로드 데이터
- 적용 대상: 중앙 관리자, Program 운영 주관 부서, 부서 담당자

이 문서의 핵심 관점은 다음과 같다.

- 어떤 데이터가 `원형`인지
- 어떤 데이터가 `연도별 공식 운영본`인지
- 어떤 데이터가 `원천값`인지
- 어떤 데이터가 `최종 공식값`인지
- 누가 무엇을 바꿀 수 있는지
- 언제부터 편집이 제한되는지

---

## 1. 데이터 계층 정책

### 1.1 기본 원칙

플랫폼 데이터는 아래 5계층으로 구분한다.

1. `시스템 공통 자산 데이터`
2. `사업 구조 데이터`
3. `연도별 운영 데이터`
4. `기간별 입력 데이터`
5. `공식 판단/보고 데이터`

### 1.2 계층별 의미

- `Program`
  - 사업 단위
  - 예: 2026 중장기발전계획, 산학협력 중장기발전계획
- `MetricLibraryItem`
  - 시스템관리에서 재사용 가능한 지표 라이브러리 항목
  - 유형: 정성 지표 / 단일 정량 지표 / 합성 정량 지표
  - 예: 학부 정원 충원율, 취업률, 교육과정 개선 추진, 교육성과 종합점수
- `DataSource`
  - 여러 사업이 공통으로 쓰는 외부 시스템 연결 자산
- `ProgramVersion`
  - 특정 연도 또는 특정 차수의 공식 운영본
  - 예: 2026 운영본, 2027 운영본
- `ProgramVersionItem`
  - 해당 운영본에서 실제 사용한 과제/지표 고정본
- `PlanRecord`, `ActualRecord`, `StatusRecord`
  - 기간별 계획, 실적, 판정 데이터
### 1.3 정책 결론

- 지표 라이브러리와 연도별 운영본은 반드시 분리한다.
- 실제 운영과 보고의 기준은 `MetricLibraryItem`이 아니라 `ProgramVersionItem`이다.
- 과거 데이터 재현은 항상 해당 연도의 `ProgramVersion` 기준으로 수행한다.

---

## 2. 생성 주체와 운영 주체 정책

### 2.1 기본 원칙

- `Program 생성 부서`와 `Program 운영 주관 부서`는 다를 수 있다.
- `Program.ownerOrgId`는 해당 Program의 기본 운영 주관 부서다.
- 운영 주관 부서는 기본적으로 해당 Program의 `program_admin` 역할을 가진다.

### 2.2 적용 예시

- 기획조정팀이 Program을 최초 생성
- 운영 주관 부서를 산학협력팀으로 지정
- 이후 산학협력팀이 분류 체계, 지표, 기간, 권한을 관리하고, 공통 상태 코드셋은 시스템관리에서 선택해 사용

### 2.3 정책 결론

- `생성 권한`과 `운영 책임`은 분리 가능하다.
- 중앙부서는 Program 개설 권한을 가질 수 있고, 운영 주관 부서는 해당 Program의 실질 관리자 역할을 수행할 수 있다.

---

## 3. 연도별 운영본 정책

### 3.1 ProgramVersion의 목적

`ProgramVersion`은 특정 연도의 공식 운영 기준을 보존하기 위한 객체다.

다음 요소는 해마다 달라질 수 있으므로 운영본으로 분리한다.

- 포함 과제/지표
- 상위 분류 위치
- 담당 부서
- 측정 방식
- 수집 방식
- 판정 방식
- 산식
- 상태셋 연결

### 3.2 운영본 생성 방식

운영본은 다음 3가지 방식으로 생성할 수 있다.

1. `빈 운영본 생성`
2. `사업 전용 템플릿에서 생성`
3. `전년도 운영본 불러오기`

### 3.2-1 운영 지표 재사용 방식

- 운영 지표는 지표 라이브러리에서 가져와 생성할 수 있다.
- 라이브러리에 없는 지표는 사업관리 화면에서 새로 만들 수 있다.
- 새로 만든 지표는 생성자, 최초 생성 사업, 최초 운영본 정보를 함께 남기며 라이브러리에 등록할 수 있다.
- 운영본이 정리되면 동일 사업 재사용용 `ProgramTemplate`의 출처가 될 수 있다.

### 3.3 정책 결론

- `템플릿`은 설계 재사용 수단이다.
- `전년도 불러오기`는 가장 자주 쓰는 운영 편의 기능이다.
- `ProgramVersion`은 해당 연도 공식 관리 대상의 보존 수단이다.

---

## 4. 최종 확정 정책

### 4.1 의미

`ProgramVersion 최종 확정(편집 불가)`은 해당 연도 운영본을 공식 기준으로 확정하고, 일반 편집을 제한하는 상태를 의미한다.

### 4.2 상태 구분

- `draft`
  - 초안 작성 중
  - 자유 수정 가능
- `active`
  - 운영 중
  - 입력 허용
  - 구조/설정 변경은 가능하나 주의 필요
- `confirmed_locked`
  - 최종 확정
  - 일반 편집 불가
- `archived`
  - 종료 후 보관

### 4.3 최종 확정 후 제한 대상

- 과제/지표 추가/삭제
- 분류 체계 내 위치 변경
- 담당 부서 변경
- 정량 산식 변경
- 수집 방식 변경
- 상태셋 변경

### 4.4 최종 확정 후 허용 예외

- 관리자에 의한 제한적 해제
- 새 revision 운영본 생성
- 설명성 메모 추가

### 4.5 정책 결론

- 최종 확정은 자동이 아니라 관리자가 선택한다.
- 최종 확정된 운영본은 해당 연도의 공식본으로 간주한다.
- 공식본 변경이 필요하면 기존 운영본을 덮어쓰기보다 새 revision 운영본 생성이 바람직하다.

## 5. 입력 및 공식값 정책

### 5.1 정량 데이터 원천 구분

정량 실적값은 아래 값을 분리해 관리한다.

- `apiValue` (v2+)
- `manualValue`
- `finalValue`
- `valueSource`
- `valueReason`

### 5.2 수집 방식 정책

**v1 (현재)**
- `manual`
  - 사용자가 직접 입력 ✅

**v2+ (후속)**
- `api`
  - 외부 시스템 연계값 사용 (외부 서버 준비 대기)
- `hybrid`
  - API 우선, 미연계/실패/예외 시 수기 입력 또는 수기 보정 허용

### 5.3 공식값 정책

**v1 (현재)**
- 화면/집계/판정의 공식 기준은 `finalValue = manualValue` (자동)
- 관리자 확정 불필요

**v2+ (추가)**
- `hybrid` 항목에서 API값과 수기값이 동시에 존재하는 경우 자동 덮어쓰지 않음
- `finalValue`는 관리자가 API값/수기값 중 선택하여 명시적으로 확정

### 5.4 정책 결론

- 원천값과 공식값은 반드시 분리한다.
- v1은 manual만 지원하므로 finalValue = manualValue (자동 결정)
- v2부터 API/hybrid 지원으로 복잡도 증가

---

## 6. 판정 정책

### 6.1 판정 구조

정량 항목은 아래 3단 구조를 사용한다.

- `autoStatus` (항상)
- `manualStatus` (v2+)
- `finalStatus`

### 6.2 정책 원칙

**v1 (현재)**
- 자동판정은 MetricFormula의 기준값에 따라 계산
- 자동판정 결과는 관리자 대시보드에서만 조회 가능 (사용자에게는 미노출)
- `finalStatus = autoStatus` (자동 결정)

**v2+ (추가)**
- 수기판정: 필요한 경우 관리자가 별도로 입력
- 최종 집계와 보고에는 `finalStatus`만 반영
- 상태 변경 사유는 `statusReason`으로 기록

### 6.3 정책 결론

- v1: 자동판정만 (내부 로직, 관리자 조회용)
- v2: 수기판정 추가로 관리자의 의사결정 반영 가능

---

## 7. 권한 정책

### 7.1 권한 부여 원칙

- 권한은 기본적으로 `부서 단위`로 부여한다.
- 예외적으로 자문위원회, 평가위원 등 외부 검토 목적에 한해 `external_reviewer`를 개인 단위로 부여할 수 있다.
- `external_reviewer`는 `Program 전체 조회`만 허용한다.
- 일반 사용자 입력 권한은 부서 단위만 허용한다.
- 개인은 입력/수정/상태변경 로그의 기록 주체로 사용한다.

### 7.2 권한 범위

- `program`
- `classification_node`

### 7.3 역할 구분

- `program_admin`
  - 구조, 기간, 권한, 지표, 운영본, 최종값 관리
- `department_editor`
  - 배정 범위 내 조회 및 입력
- `department_viewer`
  - 배정 범위 내 조회만 가능
- `external_reviewer`
  - 제한적 조회만 가능
  - Program 전체 조회만 가능
  - 입력/수정/확정 권한 없음

### 7.4 정책 결론

- 사용자는 조회+입력 가능
- 관리자는 설정+통제 가능
- 외부 검토자는 개인 단위 예외로 허용하되, 조회 전용으로 제한한다

---

## 8. 수정 가능 범위 정책

### 8.1 수정 가능 데이터

- `draft` 상태의 ProgramVersion 구조
- `active` 상태의 일반 입력 데이터
- 마감 전 계획/실적 입력값

### 8.2 제한 필요 데이터

- 최종 확정된 ProgramVersion 구조
- 이미 확정된 finalValue/finalStatus

### 8.3 권장 원칙

- 삭제보다 비활성화를 우선한다.
- 과거 실적이 있는 항목은 물리 삭제하지 않는다.
- 구조 변경은 가급적 미래 기간 또는 새 운영본에 반영한다.

---

## 9. 기간 및 마감 정책

### 9.1 기간 정책

- 기간은 관리자 정의 `typeCode` 기반으로 생성한다.
- 모든 입력은 특정 `Period`에 연결된다.
- `Period`는 특정 `ProgramVersion`에 연결된다.

### 9.2 마감 정책

- 마감 전: 일반 입력 가능
- 마감 후: 일반 사용자 수정 제한
- 마감 후 예외 수정: 관리자 승인 또는 관리자 직접 수정

### 9.3 기준일 정책

- 기준일은 Program 관리자 또는 운영 주관 부서가 설정한다.
- 정량값의 측정일과 입력/판정 기준 시점은 구분해 관리할 수 있다.

### 9.4 정책 결론

- 기간과 기준일은 명확히 분리한다.
- 마감 이후 수정 정책은 별도 예외 절차를 둔다.

---

## 10. 업로드 및 연계 정책

### 10.1 업로드 정책

- 분류 체계 업로드와 지표 업로드는 분리한다.
- 지표 업로드 파일에는 `classificationNodeCode`를 포함한다.
- 시스템은 코드 기준으로 자동 매핑한다.
- 분류 체계가 준비되지 않으면 지표 업로드는 실패 처리한다.

### 10.2 대량 업로드 정책

- 계획/실적 업로드는 `ImportBatch`로 관리한다.
- `ImportBatch`는 특정 `ProgramVersion` 기준으로 수행한다.
- 부분 실패를 허용하되 오류 내역을 제공한다.
- 최초 업로드 원본은 `ImportedSnapshot`으로 보존할 수 있다.

### 10.3 API 연계 정책

**v1 (현재)**
- API 연계 미지원 (collectionMode = manual만)

**v2+ (후속)**
- DataSource 등록과 인증 설정은 시스템관리의 `API 관리`에서 수행
- 기본 API 연결과 MetricMapping은 시스템관리의 `지표 라이브러리`에서 정의
- 사업은 라이브러리 정의를 가져와 사용하며, 별도 사업관리 메뉴는 두지 않는다.

### 10.4 정책 결론

- v1: 업로드는 검증 중심, 수기 입력 중심
- v2: API 연계 추가로 자동화 수준 향상

## 12. 이력 및 감사 정책

### 12.1 기록 대상

- 계획/실적 입력
- finalValue 변경
- finalStatus 변경
- 권한 배당/회수
- 운영본 최종 확정

### 12.2 기록 단위

- 변경자
- 변경 시각
- 변경 대상
- 변경 전 값
- 변경 후 값
- 변경 사유

### 12.3 정책 결론

- 모든 중요한 운영 변경은 감사 가능해야 한다.
- 사용자 개인은 권한 주체가 아니라 로그 주체다.

---

## 13. 증빙 정책

### 13.1 기본 원칙

- 증빙은 현재본만 관리한다.
- 증빙 이력은 설계 범위에 포함하지 않는다.

### 13.2 워크플로우와 증빙의 관계

- 제출/접수/반려가 도입되더라도 증빙 자체의 버전 이력은 관리하지 않는다.
- 워크플로우 판단 기준은 제출 시점의 현재 증빙이다.
- 반려 시에는 관리자 의견 또는 반려 사유를 기록한다.

### 13.2 구성 요소

- `evidenceNote`
- `EvidenceLink`
- `Attachment`

### 13.3 정책 결론

- 증빙은 값의 신뢰도를 보강하는 용도다.
- 첨부파일 버전관리까지는 초기 범위에 포함하지 않는다.

---

## 14. 보존 및 삭제 정책

### 14.1 보존 원칙

- 운영 데이터는 최소 5년 이상 보존을 기본 가정으로 한다.
- 종료된 Program과 ProgramVersion은 `archived` 상태로 보관한다.

### 14.2 삭제 원칙

- 구조 데이터는 물리 삭제보다 비활성화를 우선한다.
- 실적이 존재하는 항목은 삭제하지 않는다.
- 삭제가 필요한 경우 관리자만 수행하고 감사 로그를 남긴다.

### 14.3 정책 결론

- 실무 데이터는 삭제보다 보존이 우선이다.
- 사용자 관점에서는 “안 보이게 처리”하고, 시스템 관점에서는 “남겨두는 방식”이 적합하다.

---

## 15. 데이터 품질 정책

### 15.1 필수 검증 항목

- 필수값 누락 여부
- 코드값 유효성
- 데이터 타입 일치 여부
- 기간-권한 범위 일치 여부
- 정량값 허용 범위 여부

### 15.2 이상치 대응

- 급격한 증감 감지
- 허용 범위 초과 알림
- API값과 수기값 불일치 경고

### 15.3 정책 결론

- 저장 가능 여부와 데이터 신뢰도 평가는 분리해 생각한다.
- 품질 경고는 입력 방해보다 관리자 검토를 돕는 방향이 적절하다.

## 16. 입력폼 구성 정책

### 16.1 기본 원칙

- 입력 화면은 `저장 필드`와 `안내/설명 블록`을 분리해 관리한다.
- 안내 블록은 집계 대상이 아니며 화면 구성 목적의 데이터다.

### 16.2 지원 블록 예시

- `field`
- `info_card`
- `warning_card`
- `plain_text`
- `section_title`
- `divider`
- `link_block`

### 16.3 운영 원칙

- 관리자는 특정 Program 또는 ProgramVersion 기준으로 입력폼을 자유롭게 구성할 수 있다.
- 관리자는 드래그 앤 드롭으로 필드와 안내 문구 위치를 조정할 수 있다.
- 필드 수정은 인라인이 아니라 중앙 모달에서 수행하는 것이 원칙이다.
- `코드셋 선택` 필드는 기존 공통 코드셋을 선택하거나 직접 옵션을 입력할 수 있다.
- 직접 옵션 입력 후 저장 시 해당 옵션을 공통 코드셋으로 승격할지 확인하는 별도 모달을 둘 수 있다.
- 특정 지표/과제, 특정 분류 노드, 특정 입력 유형에만 블록을 조건부로 노출할 수 있다.
- 같은 블록이라도 계획 입력 기간과 실적 입력 기간에 따라 editable / read_only / hidden으로 다르게 렌더링할 수 있다.
- 실적 입력 기간에는 같은 부서가 입력한 최신 계획값 또는 계획 서술을 읽기 전용으로 함께 보여줄 수 있다.

### 16.4 정책 결론

- `입력 데이터 구조`와 `사용자 안내 구조`는 분리한다.
- 범용성 확보를 위해 입력창도 메타데이터 기반으로 관리한다.

## 17. 추후 의사결정이 필요한 항목

- ProgramVersion이 `active` 상태일 때 구조 변경 허용 범위를 어디까지 둘지
- 최종 확정 후 해제 권한을 누구에게 줄지
- finalValue 확정의 기본 주체를 Program 관리자만으로 둘지, 운영 주관 부서 관리자까지 포함할지
- 마감 후 수정 프로세스를 승인형으로 둘지 관리자 직접 수정형으로 둘지
- ImportedSnapshot을 모든 업로드에 남길지, 특정 업로드에만 남길지

## 18. 현재 기준 요약

- 권한은 부서 단위, 로그는 개인 단위
- 지표 라이브러리, API 소스, 상태 코드셋은 시스템관리 자산
- 실제 운영 기준은 `ProgramVersion / ProgramVersionItem`
- 공식 판단값은 `finalValue / finalStatus`
- 여러 지표를 하나로 합치는 요구는 별도 묶음이 아니라 `합성 정량 지표`로 관리
- 사업 전용 템플릿은 재사용용, 전년도 복사는 운영 편의용, 운영본은 공식 보존용
- 최종 확정은 관리자가 선택하는 `편집 불가` 상태
- 입력 화면은 필드와 안내 블록을 드래그 앤 드롭으로 배치하는 구조
