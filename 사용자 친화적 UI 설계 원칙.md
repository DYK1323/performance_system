## P1. 이중 입력 경로 (Dual Entry Path)

**정의**: 엑셀 업로드와 웹 인라인 편집은 기능 계층의 차이 없이 동등한 입력 수단이며, 어느 경로로 입력해도 동일한 결과물이 생성된다.

### 각 경로의 적합 상황

| 상황 | 권장 경로 | 이유 |
|------|-----------|------|
| 초기 계획값 일괄 등록 | 엑셀 업로드 | 수십~수백 개 행을 한 번에 처리, 기존 엑셀 원본 재활용 가능 |
| 분류 노드(ClassificationNode) 대량 생성 | 엑셀 업로드 | 트리 구조를 탭/열로 표현하는 것이 직관적 |
| 개별 실적값 수정 | 웹 인라인 | 단건 수정에 파일 작업이 불필요 |
| 증빙 설명(evidenceNote) 입력 | 웹 인라인 | 파일 첨부, 링크 등 복합 행위가 동반 |
| 특정 기간 전체 실적 마감 처리 | 엑셀 업로드 | 담당자가 오프라인에서 취합 후 일괄 제출 |

### 엑셀 업로드 구체 구현 원칙

1. **템플릿 강제 제공**: 업로드 전 "이번 기간 입력 가능 항목" 기준으로 시스템이 생성한 템플릿(.xlsx)을 다운로드하게 한다. 빈 셀 구조와 유효값 목록이 포함된 템플릿이어야 한다.

2. **오류 피드백 즉시, 행 단위**: 업로드 후 성공/실패를 ImportBatch 단위로 요약하되, 오류는 반드시 원본 행 번호와 함께 표시한다. 예: "3행 D열 - '달성완료'는 최종 점검 기간에는 유효하지 않은 상태값입니다."

3. **부분 성공 허용**: 오류가 있는 행을 건너뛰고 나머지 행은 저장한다. 사용자는 오류 행만 수정해서 재업로드할 수 있다.
<-- 응? 아예 업로드가 안 되는게 사용자가 잘 업로드되었다고 착각하지 않도록 해줄 수 있지 않을까? -- >

4. **업로드 이력 가시화**: ImportBatch 이력은 관리자와 업로더 모두 조회 가능하며, 언제 누가 어느 파일을 올렸는지 7일 이상 보존한다.

### 안티패턴

- 엑셀로 올린 데이터를 웹에서 수정할 수 없도록 잠금 처리하는 것
- 업로드 성공/실패를 "오류가 있습니다"라는 메시지 하나로만 표시하는 것
- 전체 행이 오류 없어야만 업로드를 허용하는 전-성공/전-실패 방식
- 템플릿 없이 항목 ID를 사용자가 직접 입력하도록 요구하는 것

## P2. 즉각 피드백 루프 (Immediate Feedback Loop)

**정의**: 사용자 행동(입력, 저장, 판정 변경)의 결과는 조작 후 500ms 이내에 화면에 반영되어야 하며, 저장 버튼 클릭 횟수를 최소화한다.

### 자동 저장 원칙

- **셀 이탈(blur) 시 자동 저장**: 웹 인라인 편집에서 셀을 떠나는 순간 자동으로 PUT 요청을 발행한다. 저장 버튼은 존재하되 "전체 저장" 확인 용도로만 쓴다.
- **저장 상태 표시**: 셀 또는 행 우측에 3단계 상태를 표시한다.
  - 회전 아이콘 + "저장 중..."
  - 체크 아이콘 + "저장됨 (14:32)"
  - 경고 아이콘 + "저장 실패 - 다시 시도"
- **달성률 즉시 갱신**: actualValue를 입력하는 즉시 클라이언트에서 `(actualValue / goalValue) * 100`을 계산해 달성률과 autoStatus를 화면에 반영한다. 서버 확인은 비동기로 수행한다.

### 오류 표시 원칙

- **인라인 오류**: 입력 규칙 위반은 해당 셀 바로 아래 붉은 텍스트로 표시한다. 모달이나 알림 팝업을 사용하지 않는다.
- **심각도 분류**:
  - 오류(저장 불가): 빨간 테두리 + 저장 차단 (예: 필수값 누락, 범위 초과)
  - 경고(저장 허용): 노란 배경 + 저장 가능 (예: 목표값이 전년도 실적보다 낮음)
  - 정보(참고용): 회색 텍스트 + 저장 가능 (예: "이전 기간 실적: 92")
- **비워도 되는 필드와 필수 필드 구분**: 필수 필드는 레이블에 `*` 표시 및 빈 상태에서 저장 시도 시 즉시 강조.

### 동시성 충돌 알림

낙관적 잠금(version 필드 기반) 충돌 발생 시:
- "다른 사용자(홍길동, 14:31)가 이 항목을 수정했습니다. 현재 값을 확인하고 다시 입력해 주세요."라는 비침습적 배너(화면 상단 슬라이드인)를 표시한다.
- 사용자가 입력한 값은 폼 안에 유지한 채로 최신 서버값을 읽기 전용으로 나란히 표시한다.
- 사용자는 "내 값으로 덮어쓰기" 또는 "서버값 사용" 중 선택한다.

### 안티패턴

- 모든 필드를 입력하고 맨 아래 "저장" 버튼을 눌러야만 반영되는 방식
- 충돌 발생 시 사용자의 입력 내용을 조용히 덮어쓰는 것
- 오류 메시지를 페이지 상단 알림 하나로 취합해서 어느 셀인지 알 수 없게 만드는 것
- 달성률을 페이지 새로고침 후에야 갱신하는 것

## P3. 담당 항목 우선 노출 (Owner-First Navigation)

**정의**: 부서 담당자는 로그인 직후 자신이 입력해야 할 항목에 최소한의 탐색 단계로 도달해야 하며, 미입력 항목은 완료 항목보다 항상 우선적으로 노출된다.

### 진입 경로 설계

1. **로그인 → 대시보드**: SSO 조직 정보를 기반으로 해당 담당자의 미입력 항목 수를 즉시 표시한다. "이번 기간 미입력 항목 7건" 카드를 클릭하면 바로 해당 목록으로 이동한다.

2. **기본 필터 자동 적용**: 항목 목록 진입 시 `ownerId = 현재 사용자 조직`, `period = 현재 활성 기간`, `status = 미입력` 필터가 기본값으로 적용되어 있다. 사용자가 명시적으로 필터를 해제해야만 다른 항목이 보인다.

3. **계층 트리 탐색**: 항목이 수백 개인 경우 2~4단계 계층 구조를 좌측 트리 패널로 표시한다. 트리 노드에 미입력 수 배지(badge)를 표시해 어느 분류에 미입력이 남았는지 한눈에 파악 가능하게 한다.

### 미입력 항목 강조 원칙

- **목록 내 정렬 기본값**: 미입력 항목이 상단, 입력 완료 항목이 하단. 마감일 임박 항목은 미입력 그룹 내에서 추가로 상단 정렬.
- **시각적 강조**: 미입력 행은 배경색 없음 + 좌측 3px 주황 테두리. 입력 완료 행은 배경 연회색으로 시각적 후퇴.
- **마감일 카운트다운**: 기간 마감 D-3 이하인 미입력 항목에는 "마감 2일 전" 레이블을 행 내에 인라인 표시.

### 검색 설계

- 항목명, 분류 코드, 담당 부서명으로 실시간 검색(타이핑 300ms 후 결과 필터링).
- 검색어 하이라이팅: 일치하는 텍스트는 굵은 글씨 + 노란 배경으로 표시.
- 계층 트리와 목록이 동기화: 트리에서 노드를 클릭하면 해당 분류의 항목만 목록에 표시된다.

### 안티패턴

- 로그인 후 전체 항목 목록을 필터 없이 표시하는 것
- 입력 완료 항목과 미입력 항목이 섞여서 정렬되는 것
- 계층 구조가 드롭다운 하나로 압축되어 상하위 관계가 보이지 않는 것
- 자신 담당 항목을 찾기 위해 부서 필터를 매번 수동으로 설정해야 하는 것

## P4. 판정 투명성 (Judgment Transparency)

**정의**: 자동 판정(autoStatus)과 수기 판정(manualStatus)의 출처, 근거, 선택 과정은 언제나 화면에서 추적 가능해야 하며, 수기 판정 선택 시 마찰을 최소화한다.

### 판정 UX 구조

판정 영역은 하나의 행 내에 세 개의 시각 요소로 구성된다:

```
[ 자동 판정: 달성예상 (95%) ]  →  [ 수기 판정: 이슈 ▾ ]  →  [ 최종: 이슈 (수기) ]
```

- **자동 판정 영역**: 읽기 전용. MetricFormula 연산 결과와 달성률 근거를 함께 표시. 툴팁으로 판정 규칙을 확인 가능: "판정 기준: 달성률 80% 이상 = 달성예상, 100% 이상 = 달성완료"
- **수기 판정 영역**: 드롭다운 선택. StatusSet에 정의된 상태 목록만 표시. 상태 변경 시 사유 입력 모달이 슬라이드인(기본 높이 2행 textarea, 선택 아닌 권장).
- **최종 판정 영역**: autoStatus 또는 manualStatus 중 어느 것이 적용되는지 출처 라벨과 함께 표시. 수기 판정이 없으면 자동 판정이 최종값이 된다.

### 수기 판정 선택 플로우

```
[수기 판정 드롭다운 열기]
  → 현재 autoStatus가 상단에 "자동 계산됨: 달성예상" 으로 표시
  → 사용자가 다른 상태 선택 (예: "이슈")
  → 인라인 사유 입력창 등장 (2행 텍스트 박스)
  → 사유 입력 후 확인 → 저장
  → 최종 판정이 "이슈 (수기)"로 변경, statusChangedAt 기록
```

### 판정 이력 표시

- 각 항목의 "판정 이력" 아이콘(시계 모양)을 클릭하면 statusReason 포함 변경 로그를 시간순으로 표시한다.
- 이력 항목: "2026-04-09 14:32 | 홍길동 | 달성예상 → 이슈 | 사유: 주관부서 통계 지연"

### 정성 항목의 판정

정성 항목은 autoStatus가 없으므로 수기 판정만 존재한다. 이 경우:
- 자동 판정 영역은 "자동 판정 없음 (정성 항목)"으로 표시하고 회색 처리한다.
- 수기 판정 드롭다운과 최종 판정 영역은 정량 항목과 동일한 위치에 표시한다.

### 안티패턴

- 자동 판정과 수기 판정이 같은 드롭다운에 섞여 있어 어느 것이 시스템 계산인지 알 수 없는 것
- 수기 판정을 입력했는데 화면이 갱신되면 자동 판정이 최종값으로 덮어쓰이는 것
- 판정 변경 사유 입력창을 모달로 띄워 입력 맥락을 잃게 만드는 것
- 이전에 어떤 판정을 했는지 이력이 없는 것

## P5. 단일 시각 언어 (Unified Visual Language)

**정의**: 상태 표현에 쓰이는 색상, 아이콘, 텍스트 레이블은 플랫폼 전체에서 일관되게 사용하며, 세 요소는 항상 함께 표시한다. 색상 팔레트는 플랫폼이 제공하되, 관리자가 상태명과 색상을 자유롭게 구성할 수 있다.

### 색상 팔레트 (플랫폼 기본 제공)

플랫폼에서 제공하는 12가지 색상 조합 (모두 WCAG AA 대비 비율 만족):

| 번호 | 이름 | 배경 | 텍스트 | 추천 용도 |
|------|------|------|--------|---------|
| 1 | 초록 | #E8F5E9 | #2E7D32 | 성공/달성/완료 |
| 2 | 파랑 | #E3F2FD | #1565C0 | 진행중/예상/향상 |
| 3 | 주황 | #FFF3E0 | #E65100 | 주의/미달성예상 |
| 4 | 빨강 | #FCE4EC | #B71C1C | 이슈/위험 |
| 5 | 회색 | #F5F5F5 | #757575 | 미입력/중립 |
| 6 | 보라 | #F3E5F5 | #6A1B9A | 우수/뛰어남 |
| 7 | 남색 | #E0E3FF | #283593 | 중요/핵심 |
| 8 | 청록 | #E0F2F1 | #00695C | 승인됨 |
| 9 | 분홍 | #FCE4EC | #C2185B | 주목할만함 |
| 10 | 갈색 | #EFEBE9 | #5D4037 | 검토중 |
| 11 | 짙은회색 | #EEEEEE | #424242 | 차단됨 |
| 12 | 연파랑 | #E1F5FE | #01579B | 대기중 |

### 관리자의 상태 구성 자유도

**StatusSet 관리 화면에서**:
- 상태명은 자유롭게 정의 가능 (예: "달성완료" → "목표달성", "On Track" 등)
- 색상은 위의 팔레트에서 선택하여 배치 (중복 색상 불가)
- 각 색상 선택 시 대비 비율이 자동으로 WCAG AA를 만족함을 보장
- 색상 선택 시 실시간으로 배지 미리보기를 화면에 표시

**예시**:
```
Program: 중장기발전계획 > 중간 점검
  [상태 1] 명칭: "목표달성" | 색상: 초록 (#E8F5E9 / #2E7D32) | 아이콘: check_circle
  [상태 2] 명칭: "진행중" | 색상: 파랑 (#E3F2FD / #1565C0) | 아이콘: trending_up
  [상태 3] 명칭: "위험" | 색상: 빨강 (#FCE4EC / #B71C1C) | 아이콘: error

Program: 자체평가 > 최종 평가
  [상태 1] 명칭: "A등급" | 색상: 초록
  [상태 2] 명칭: "B등급" | 색상: 파랑
  [상태 3] 명칭: "C등급" | 색상: 주황
```

### 플랫폼 전체 일관성 보장

1. **StatusSet별 독립 관리**: 각 Program 또는 기간 유형(interim/final)별로 다른 상태를 정의 가능
   - 중간 점검: 4~5개 상태
   - 최종 점검: 2~3개 상태
   - 자유롭게 구성하되, 과도하게 많은 상태(7개 이상)는 주의 필요

2. **색상 선택 UI**:
   - 관리자가 상태를 추가할 때 "색상 선택" 버튼 클릭
   - 12개 색상 팔레트가 그리드로 표시 (각 색상에 미리보기 배지)
   - 이미 사용 중인 색상은 회색 처리로 중복 방지
   - 선택한 색상이 실시간으로 현재 상태의 배지에 적용되어 미리보기됨

3. **색상 변경 시 영향도 안내**: 관리자가 색상을 변경하려고 하면
   - "이 상태 색상을 변경하면 다음에 영향을 줍니다"
   - "- 대시보드 43개 화면"
   - "- 실적값 목록 내 배지 123개"
   - "- 이력 기록 456개"
   - "변경하시겠습니까?" 다이얼로그 표시 후 확인 필요

### 시각 언어 적용 규칙

1. **상태 배지(badge)**: 색상 배경 위에 아이콘 + 텍스트 레이블을 조합한 칩(chip) 형태. 최소 너비 80px. 목록 행에서는 고정 너비 열로 배치.

2. **진행률 표시**: 기간 전체 입력률을 프로그레스 바로 표시. 색상은 입력 완료(초록 #E8F5E9/#2E7D32), 미입력(회색 #F5F5F5/#757575)으로만 구성. 달성률과 입력률을 혼용하지 않는다.

3. **대시보드 집계**: 도넛 차트, 스택 막대에서 StatusSet에 정의된 색상을 그대로 사용한다. 임의로 다른 색을 쓰지 않는다.

4. **강조 경보**: 마감 D-3 이하이면서 미입력인 항목은 "마감 임박" 레이블을 주황(#FFF3E0 / #E65100) 배경 + 느낌표 아이콘으로 추가한다. 상태 배지를 교체하지 않고, 별도 레이블을 추가한다.

5. **색상은 StatusSet이 관리**: 관리자가 StatusSet에서 색상을 변경하면 플랫폼 전체의 해당 상태 색상이 실시간으로 일괄 변경된다. 코드에 색상 하드코딩 금지.

### 안티패턴

- 같은 "달성예상"을 한 화면에서는 파란 배지로, 다른 화면에서는 초록 배지로 표시하는 것
- 아이콘만 있고 텍스트 레이블이 없는 상태 표시
- 색상이 팔레트 외 임의의 색상으로 추가되는 것 (관리자도 팔레트 내에서만 선택 가능해야 함)
- 입력률(%)과 달성률(%)을 같은 프로그레스 바로 섞어 표시하는 것
- StatusSet 색상 변경 시 영향도를 안내하지 않고 바로 변경되는 것

## P6. 역할 분리된 인지 부하 (Role-Separated Cognitive Load)

**정의**: 중앙 관리자 화면과 부서 담당자 화면은 메뉴 구조, 정보 밀도, 허용 행동이 완전히 다르게 설계되며, 두 역할의 사용자가 서로의 화면에 혼입되지 않는다.

### 관리자 화면 설계 원칙

**대상**: 플랫폼의 구조를 설정하는 사람 (Program, Schema, Field, StatusSet 구성)

- **정보 밀도**: 높음. 좌측 트리 + 우측 폼 구조. 설정 항목이 많으므로 탭과 아코디언을 활용하되, 탭은 최대 5개 이내.
- **액션 중심 레이아웃**: "Program 생성", "스키마 편집", "기간 추가" 등 실행 가능한 행동이 항상 상단에 가시화.
- **위험 행동 보호**: Schema 변경, StatusSet 수정 등 하위 데이터에 연쇄 영향을 미치는 변경은 영향 범위 미리 보기 후 확인 다이얼로그를 거친다. 예: "이 스키마를 변경하면 연결된 항목 43개의 표시 구조가 바뀝니다."
- **읽기 전용 미리 보기**: 담당자에게 보이는 입력 화면을 미리 보기로 확인 가능한 "담당자 뷰 미리 보기" 버튼 제공.
- **ImportBatch 대시보드**: 업로드 이력, 성공/실패율, 오류 로그를 관리자만 볼 수 있는 별도 섹션으로 분리.

### 부서 담당자 화면 설계 원칙

**대상**: 계획값·실적값을 입력하는 사람

- **정보 밀도**: 낮음. 한 화면에 보이는 항목은 최대 20~30행. 더 많으면 가상 스크롤 또는 분류 단위로 페이지 분리.
- **구조 설정 UI 완전 차단**: Schema, StatusSet, FieldDefinition을 수정하는 UI는 메뉴에 존재하지 않는다. 잘못 눌러도 접근 불가.
- **컨텍스트 항상 표시**: 현재 선택된 Program명, 기간명, 마감일이 화면 상단 고정 헤더에 항상 표시된다.
- **지표 해석 도움**: 각 항목 행에 도움말(?) 아이콘을 배치, 클릭 시 description, 산식(criteria), 이전 기간 실적, 입력 예시를 슬라이드 패널로 표시.
- **내 담당 vs 전체**: 기본은 "내 담당 항목"만 표시. "전체 보기" 토글을 제공하되 읽기 전용으로만 열람 가능.

### 네비게이션 분리

```
관리자 GNB:
  [Program 관리] [스키마 구성] [항목 설정] [기간 관리] [시스템 설정]

담당자 GNB:
  [내 현황] [입력하기] [이력 조회]
```

두 GNB는 로그인 역할에 따라 완전히 다른 것을 보여준다. 역할 전환(관리자가 담당자 입력 화면을 직접 볼 때)은 명시적인 "담당자 모드로 보기" 버튼을 통해서만 가능하며, 전환 사실을 화면 상단 배너로 표시한다.

### 안티패턴

- 담당자 입력 화면에 "스키마 편집" 버튼이 숨겨진 메뉴로라도 존재하는 것
- 관리자가 담당자 화면을 구분 없이 함께 사용하도록 설계된 단일 메뉴 구조
- 역할이 다른데 같은 레이아웃·밀도의 화면을 보여주는 것
- 관리자 설정 변경이 사전 경고 없이 즉시 담당자 화면에 반영되는 것

## P7. 모바일 확인 전용 (Mobile Read-First)

**정의**: 모바일은 입력 기능이 아닌 현황 확인에 최적화한다. 데스크톱에서 가능한 모든 입력 기능을 모바일에서 제공하려 하지 않는다.

### 모바일 지원 범위

| 기능 | 모바일 지원 여부 | 비고 |
|------|----------------|------|
| 대시보드 위젯 조회 | 지원 | 카드형 세로 스택 레이아웃 |
| 미입력 항목 현황 확인 | 지원 | 숫자 카드 + 목록 |
| 단순 텍스트/숫자 단건 입력 | 제한 지원 | 1항목씩 전체 화면 폼 |
| 판정 선택 (수기 판정) | 제한 지원 | 전체 화면 드로어 |
| 엑셀 업로드 | 미지원 | 데스크톱 권장 안내 |
| 증빙 파일 첨부 | 제한 지원 | 카메라 촬영 → 업로드 |
| 계층 트리 탐색 | 미지원 | 단일 드롭다운으로 대체 |
| 관리자 스키마 설정 | 미지원 | 데스크톱 전용 안내 |

### 모바일 레이아웃 원칙

- 테이블을 모바일에서 그대로 보여주지 않는다. 행을 카드(card)로 변환하여 세로 스크롤 목록으로 표시한다.
- 하단 고정 탭바: [내 현황], [입력], [이력] 3개만.
- 터치 타깃 최소 44x44dp 유지.
- 단건 입력 시 전체 화면 폼을 사용하고, 완료 후 목록으로 복귀.

### 안티패턴

- 데스크톱 테이블을 모바일에서 가로 스크롤로 그대로 보여주는 것
- 모바일 대응을 responsive CSS만으로 처리해 실제로 사용하기 어려운 화면을 만드는 것
- 모바일에서 엑셀 업로드 버튼을 보여주되 동작하지 않게 하는 것

## P8. 색상 없이도 동작 (Color-Independent Accessibility)

**정의**: 상태·경고·오류 등 모든 정보 표현은 색상이 유일한 구분 수단이 되어서는 안 되며, 키보드만으로 모든 핵심 기능에 접근 가능해야 한다.

### 색상 외 구분 원칙

- 상태 배지는 항상 아이콘 + 텍스트를 색상과 함께 표시한다.
- 오류 상태 셀은 붉은 테두리 + 오류 아이콘 + 오류 텍스트를 함께 사용한다.
- 달성/미달성 구분은 색상 외에 텍스트 레이블, 아이콘 방향(화살표 상향/하향)을 추가한다.
- 차트에서는 색상 외에 패턴(해칭) 또는 직접 레이블을 함께 사용한다.

### 키보드 탐색 원칙

- **Tab 순서**: 좌측 트리 → 목록 필터 → 목록 행 → 셀 편집 순서로 논리적 Tab 순서를 유지한다.
- **셀 편집**: 목록에서 항목 행에 포커스 후 Enter 키로 편집 모드 진입. Escape로 편집 취소. Tab으로 다음 편집 가능 셀 이동.
- **판정 드롭다운**: 방향키(↑↓)로 상태 선택, Enter로 확정, Escape로 취소.
- **모달/패널**: 열릴 때 포커스가 패널 내부 첫 요소로 이동. Escape로 닫힘. 닫힐 때 열기 트리거 버튼으로 포커스 복귀.
- **단축키 제공**: `G + I` = 입력 화면으로 이동, `G + D` = 대시보드로 이동 (Gmail 스타일 시퀀스 단축키, 화면 하단 `?` 버튼으로 목록 확인 가능).

### ARIA 및 스크린 리더 대응

- 자동 저장 중일 때 `aria-live="polite"` 영역에 "저장 중" 텍스트를 갱신한다. 저장 완료 시 "저장되었습니다"로 갱신.
- 상태 배지는 `role="status"` 또는 `aria-label="상태: 달성예상"`을 포함한다.
- 충돌 경고 배너는 `aria-live="assertive"`로 즉시 읽힌다.
- 테이블 헤더는 `scope="col"` 속성을 포함하고, 정렬 가능한 컬럼은 `aria-sort`를 갱신한다.

### 대비 비율 기준

- 일반 텍스트: WCAG AA 기준 4.5:1 이상
- 대형 텍스트(18pt 이상): 3:1 이상
- 상태 배지 텍스트: 배경색과 4.5:1 이상 (StatusSet 색상 정의 시 관리자 도구에서 대비 비율 자동 표시)

### 안티패턴

- "빨간 셀이 오류입니다"라고 문서에만 명시하고 스크린 리더 사용자는 알 수 없는 구조
- 마우스 호버로만 접근 가능한 판정 드롭다운
- Tab 순서가 시각적 레이아웃 순서와 다른 것
- 포커스 링(focus ring)을 CSS로 제거한 것

## 원칙 간 우선순위 충돌 해소 가이드

실제 구현 중 원칙이 서로 충돌할 때의 결정 기준:

| 충돌 상황 | 우선 원칙 | 이유 |
|-----------|-----------|------|
| 즉시 저장(P2)이 충돌 감지(P2) 보다 빠름 | 충돌 감지 후 저장 | 덮어쓰기는 데이터 손실 |
| 미입력 강조(P3)가 접근성 색상 기준(P8) 위반 | P8 우선 | 대비 비율은 비협상 기준 |
| 모바일 단순화(P7)가 역할 분리(P6) 축소로 이어짐 | P6 유지 + P7 범위 축소 | 역할 분리는 보안 경계 |
| 판정 투명성(P4)이 화면을 복잡하게 만드는 경우 | P4 유지 + 기본 접힘 처리 | 판정 출처 추적은 감사 요건 |

## 구현 체크리스트

개발·디자인 단계에서 각 화면을 검토할 때 사용하는 빠른 체크리스트:

### 입력 관련
- [ ] 셀 이탈 시 자동 저장이 동작하는가
- [ ] 저장 상태(저장 중/완료/실패)가 화면에 표시되는가
- [ ] 엑셀 업로드 오류가 행 번호와 함께 표시되는가
- [ ] 엑셀 업로드 전 템플릿 다운로드 과정이 있는가
- [ ] 엑셀로 올린 데이터도 웹에서 수정 가능한가

### 탐색 관련
- [ ] 로그인 후 미입력 항목에 3클릭 이내 도달 가능한가
- [ ] 기본 필터가 "내 담당 + 현재 기간 + 미입력"으로 설정되어 있는가
- [ ] 계층 트리에 미입력 수 배지가 표시되는가
- [ ] 마감 D-3 이하 항목에 "마감 임박" 표시가 있는가
- [ ] 검색어 하이라이팅이 구현되어 있는가

### 판정 관련
- [ ] 자동 판정과 수기 판정이 시각적으로 구분되는가
- [ ] 최종 판정에 출처(auto/manual)가 표시되는가
- [ ] 판정 변경 이력을 조회할 수 있는가
- [ ] 판정 변경 시 사유 입력이 가능한가
- [ ] 정성 항목에는 자동 판정 영역이 회색 처리되는가

### 시각 언어 관련
- [ ] 동일 상태가 플랫폼 전체에서 같은 색상·아이콘·레이블을 쓰는가
- [ ] 상태 표시에 색상만 사용하지 않았는가 (아이콘+텍스트 동반)
- [ ] 색상은 StatusSet에서 관리되고 코드에 하드코딩되지 않았는가
- [ ] 진행률(입력률)과 달성률이 구분되는가
