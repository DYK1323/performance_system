# 범용 성과계획·실적 관리 플랫폼 기획안 v2

## Summary

- 목표: `발전계획 전용 시스템`이 아니라, 기획조정팀이 여러 사업·평가·재정지원 과제에서 공통으로 쓸 수 있는 `메타데이터 기반 성과관리 플랫폼`
- v2 중심: 개별 사업 화면이 아니라 `관리자가 관리 대단위를 정의하고`, 구조·항목·기간·수집방식·집계방식·시각화를 **명확한 규칙** 하에 설정
- 기준 정의 구조: `Program > ClassificationSchema > ManagedItem(quantitative|qualitative)`
- 실제 운영 구조: `Program > ProgramVersion > ProgramVersionItem`
- 초기 주요 사용자: `중앙 관리자` 및 `부서 담당자` (SSO 기반 인증)
- 입력 흐름: `입력 후 즉시 반영` (제출-검토-확정은 v2 이후 확장)
- 기간: `관리자 정의형` (연도, 학기, 분기, 특정 차수 자유 구성)
- 범용성 원칙: 최소 관리 단위 타입은 `정량/정성` 2종으로 단순화하되, 정량 내부의 `측정 방식`과 `판정 방식`은 별도 속성으로 분리
- 운영 주체 원칙: `Program 생성 주체`와 `Program 운영 주체`는 분리 가능하며, 운영 주체 부서는 해당 Program의 기본 관리자 역할을 가진다.
- 재사용 원칙: 같은 사업 안에서 연도별 과제/지표/산식이 달라질 수 있으므로 `Template`, `전년도 불러오기`, `연도별 운영본(ProgramVersion)`을 함께 지원한다.
- 기획 목적: 단순 수집 화면이 아니라 `계획-실적-점검-환류`를 연결하는 운영 플랫폼
- **새 추가**: 동시성 제어, 상태 전환 규칙, 성능 기준, UI 화면 정의

## 1. 핵심 설계 원칙

### 1.1 데이터 식별
- 모든 엔티티는 **불변 ID** + **인간친화적 명칭** 분리 관리
- 명칭은 언제든 변경 가능, ID는 모든 레코드에서 영구 참조
- 예: `program_2026_roadmap`(ID) vs "2026 중장기발전계획"(명칭)

### 1.2 계층 구조
- **ClassificationSchema**: 관리자가 정의한 분류 레벨 구조
  - 예1) 레벨 2: `영역 > 전략 > [최소관리단위]`
  - 예2) 레벨 4: `사업군 > 세부사업 > 세부과제 > [최소관리단위]`
  - 예3) 무분류: 모든 항목이 Program 직하위
- 각 ClassificationNode는 **단일 부모만 가짐** (트리 구조)
- 최종 관리 대상은 항상 **최하위 노드의 ManagedItem**

### 1.3 이원화된 데이터 흐름
- **기간(Period)**: 관리 단위 (예: "2026-1차 점검기간", "2026-1학기")
  - Program 아래 관리자 생성
  - 계획값/실적값 입력 시 반드시 기간 선택
  
- **점검 주기(MonitoringCycle)**: 항목별 속성
  - 예: 월별, 분기별, 연도별, 수시, 특정기간 연동
  - 같은 기간이라도 항목마다 다른 주기 가능
  - 기간과 주기의 교집합 구간에만 입력 가능

### 1.4 판정 체계 (정량 항목)
```
입력값 → [자동 판정 규칙 적용] → autoStatus
        → [사용자 수동 판정] → manualStatus
        → [최종값 결정] → finalStatus (화면/집계에 반영)
        → 변경 시 statusReason 기록
```

- **autoStatus**: MetricFormula의 판정식으로 자동 계산
  - 예: `(actual / goal) * 100`이 85 이상이면 "달성예상"
- **manualStatus**: 사용자가 직접 선택 (필드 속성은 StatusSet으로 정의)
- **finalStatus**: autoStatus 또는 manualStatus 중 선택, 출처 기록
- **statusReason**: 변경 사유/비고 (255자 텍스트)

### 1.5 정량 항목의 하위 유형
- 최소 관리 단위 타입은 `quantitative | qualitative`만 사용한다.
- 다만 정량 항목은 실제 판정 안정성을 위해 하위 속성을 반드시 가진다.
  - `measurementType`: `number | ratio | boolean | range | composite`
  - `judgmentMode`: `higher_better | lower_better | target_range | formula_based | manual_only`
  - `collectionMode`: `manual | api | hybrid`
- 예시
  - `number + higher_better`: 논문 수, 참여 인원
  - `ratio + higher_better`: 충원율, 취업률
  - `boolean + manual_only`: 제도 도입 여부, 인증 획득 여부
  - `range + target_range`: 유지율 70~80, 부채비율 50 이하
  - `composite + formula_based`: 여러 원천값을 조합해 산출하는 지표
- 수집 방식 예시
  - `manual`: API 연결 없이 담당 부서가 기준일에 직접 입력
  - `api`: 외부 시스템 연계값을 기본으로 사용
  - `hybrid`: API 값을 우선 사용하되, 미연계/실패/예외 시 수기 입력 또는 수기 보정 허용
- 복수 지표형은 별도 최소 관리 단위 타입으로 만들지 않는다.
  - 대신 상위 분류 노드 또는 항목 묶음(Group) 아래 여러 정량 항목을 배치하고, 상위에서는 묶음 조회/집계만 제공한다.
  - 예: `취업률` 묶음 아래 `전체`, `남`, `여`, `학과별` 세부 항목을 둘 수 있다.

### 1.6 권한 원칙
- 인증은 SSO를 기본으로 하되, 권한은 SSO 조직만으로 고정하지 않는다.
- 권한은 `ProgramRoleBinding`으로 별도 관리한다.
  - 예: `program_admin`, `department_editor`, `department_viewer`
- 권한 부여 대상은 사용자 개인이 아니라 `부서`다.
- 관리자는 부서별로 `프로그램 전체` 또는 `특정 분류 노드` 범위의 입력/조회 권한을 부여한다.
- 일반 사용자는 부여된 범위 안에서 실적 입력과 조회를 수행하고, 관리자만 설정을 변경한다.
- `ownerOrgId`로 지정된 운영 주관 부서는 기본적으로 해당 Program의 `program_admin` 역할을 가진다.
- Program을 최초 생성한 부서와 실제 운영하는 부서는 서로 다를 수 있다.

## 2. 데이터 모델 상세

### 2.1 핵심 객체 정의

#### Program
```typescript
{
  id: string                    // "prog_2026_roadmap"
  name: string                  // "2026 중장기발전계획"
  description: string           // 설명
  status: "active" | "inactive"
  createdByOrgId: string        // 최초 생성 부서
  ownerOrgId: string            // 운영 주관 부서
  createdAt: timestamp
  createdBy: SSO_ID
  updatedAt: timestamp
  updatedBy: SSO_ID
}
```

#### ProgramVersion
```typescript
{
  id: string                    // "progver_2026_roadmap_main"
  programId: string             // Program 참조
  name: string                  // "2026 운영본"
  yearLabel: string             // "2026"
  versionType: "annual" | "special" | "revision"
  sourceType: "empty" | "template" | "previous_version"
  sourceTemplateId?: string
  sourceProgramVersionId?: string
  status: "draft" | "active" | "confirmed_locked" | "archived"

```

#### ClassificationSchema
```typescript
{
  id: string                    // "schema_roadmap_v1"

    sortOrder: number
```

```typescript

  name: string                  // "교육혁신"
```

#### ManagedItem
```typescript
{
  id: string                    // "item_001"
  programId: string
  name: string                  // "학부 정원 충원율"
  type: "quantitative" | "qualitative"
  
  // 공통
  description: string
  defaultOwnerOrgId?: string    // 기본 담당 부서
  monitoringCycleId: string     // MonitoringCycle 참조
  
  // quantitative 전용

  measurementType?: "number" | "ratio" | "boolean" | "range" | "composite"

  judgmentMode?: "higher_better" | "lower_better" | "target_range" | "formula_based" | "manual_only"
  collectionMode?: "manual" | "api" | "hybrid"
  metricFormulaId?: string      // MetricFormula 참조
  
  // qualitative 전용
  contentFields?: string[]      // [필드ID, 필드ID] - 정성 항목 구성 필드
  
  status: "active" | "inactive"
  createdAt: timestamp
  createdBy: SSO_ID
}
```

#### ProgramVersionItem

#### MetricFormula
```typescript
{
  id: string                    // "formula_enrollment_rate"
  itemId?: string               // ManagedItem 참조 (원형)
  programVersionItemId?: string // ProgramVersionItem 참조 (운영본)
  
  // 자동 판정 규칙
  autoJudgmentRules: {
    expression: string          // "(actual / goal) * 100"
    thresholds: {
      status: string            // "달성완료", "달성예상", "미달성예상"
      min?: number
      max?: number
      operator?: ">=" | "<=" | "==" | "between"
    }[]
  }
  
  // range/boolean/composite 대응
  targetRange?: {
    min?: number
    max?: number
    inclusive?: boolean
  }
  booleanTarget?: "Y" | "N" | true | false
  
  // 결측치 처리
  missingValuePolicy: "reject" | "useDefault" | "interpolate"
  missingValueDefault?: number
  
  createdAt: timestamp
  createdBy: SSO_ID
}
```

```typescript
{

    name: string                // "달성완료" (관리자 정의 가능)
```

### 2.2 템플릿, 전년도 불러오기, 연도별 운영본의 관계

- `ItemTemplate`는 반복 생성용 설계 원형이다.
- `FormBlockDefinition`은 입력폼을 구성하는 화면 블록 원형이다.
  - 예: 안내 문구, warning card, plain text, section title, 실제 입력 필드
- `ProgramFormLayout`은 특정 Program 또는 ProgramVersion에서 어떤 블록을 어떤 순서로 보여줄지 정의한다.
- `ProgramFormLayoutBlock`은 개별 블록의 배치 순서와 적용 범위를 정의한다.
- `ManagedItem`는 사업 내에서 재사용 가능한 기준 항목 풀이다.
- `ProgramVersion`은 특정 연도 또는 특정 차수의 공식 운영본이다.
- `ProgramVersionItem`은 해당 운영본에서 실제로 사용한 과제/지표의 고정본이다.
- 새 연도 시작 시 운영 방식은 3가지를 지원한다.
  - 빈 운영본에서 시작
  - 템플릿에서 생성
  - 전년도 ProgramVersion 복사 후 수정
- 같은 ManagedItem을 재사용하더라도 연도별로 `소속 분류`, `담당 부서`, `산식`, `수집 방식`, `상태셋`, `사용 여부`가 달라질 수 있다.
- 같은 입력 화면이라도 연도별 또는 지표별로 `안내 문구`, `warning card`, `plain text`, `필드 순서`가 달라질 수 있다.
- 따라서 계획/실적/상태/증빙은 base ManagedItem이 아니라 해당 연도 `ProgramVersionItem`에 연결한다.

## 3. 상태 전환 및 동시성 제어

### 3.1 상태 전환 규칙

v1 기본값: **모든 상태 간 전환 가능** (역변경 포함)
- 예: "달성완료" → "미달성예상" → "달성완료" 허용
- statusReason에 변경 사유 기록

상태 변경 권한: 
- **정량 항목의 manualStatus**: `ownerOrgId` 또는 Program 관리자
- **정성 항목의 상태**: `ownerOrgId` 또는 부서 담당자
- **finalStatus**: autoStatus → manualStatus 중 선택만 (상태 수정은 아님)

### 3.2 동시성 제어

**낙관적 잠금(Optimistic Locking)** 방식 도입:
```typescript
// ActualRecord와 PlanRecord에 version 필드 추가
// 저장 시:
// 1. 클라이언트가 현재 version 함께 전송
// 2. DB: version 검증
//    - 일치하면: 저장 + version++
//    - 불일치하면: 409 Conflict 반환
// 3. 클라이언트: 최신값 다시 로드 후 재입력

// 충돌 시 UI: "다른 사용자가 수정했습니다. 최신값을 로드합니다"
```

**동시 입력 정책**:
- ActualRecord 저장 성공 시, 관련 집계 테이블 **트랜잭션 내에서** 함께 업데이트
- API 연계 데이터는 수기 입력과 **독립적 열로 관리** (충돌 없음)
- API 미연계 항목은 `collectionMode=manual`로 처리한다.
- API 실패 또는 예외 가능성이 있는 항목은 `collectionMode=hybrid`로 두고 수기 대체 입력을 허용한다.
- 정량 실적의 공식 판단값은 항상 `finalValue` 기준으로 계산하되, 원천값(`apiValue`, `manualValue`)은 함께 보존한다.

## 4. 성능 기준 및 설계

### 4.1 성능 목표

| 작업 | 목표 | 비고 |
|------|------|------|
| 로그인 (SSO) | < 1초 | 학교 SSO 서버 성능에 의존 |
| 항목 목록 조회 (필터 포함) | < 2초 | 1,000개 항목 기준 |
| 기간별 입력 폼 렌더링 | < 1.5초 | 필드 30개, 커스텀 계산식 포함 |
| 대시보드 위젯 로드 | < 3초 | 모든 위젯 병렬 로드 |
| 대량 업로드 (1,000개 행) | < 30초 | ImportBatch 생성 + 검증 포함 |
| 수기 입력 저장 | < 500ms | 동시성 제어 포함 |

### 4.2 캐싱 전략

**Redis 캐싱**:
- `program:${programId}:metadata` → 1시간 TTL (Program, Schema, Field 정의)
- `period:${programId}:active` → 24시간 TTL (활성 기간 목록)
- `item:${itemId}:current` → 5분 TTL (최신 record snapshot)

**DB 인덱싱**:
```sql
-- ActualRecord
CREATE INDEX idx_actual_item_period ON actual_records(item_id, period_id);
CREATE INDEX idx_actual_period_program ON actual_records(period_id, program_id);

-- StatusRecord
CREATE INDEX idx_status_record_actual ON status_records(actual_record_id);

-- ClassificationNode
CREATE INDEX idx_node_schema_level ON classification_nodes(schema_id, level_id);

-- 조직별 항목 조회 (SSO 조직 필터링)
CREATE INDEX idx_managed_item_owner ON managed_items(default_owner_org_id, program_id);
CREATE INDEX idx_program_version_item_owner ON program_version_items(owner_org_id, program_version_id);
```

```
  │   ├─ 실적값 입력 (정량: 달성값, 정성: 추진현황+리스크)
  │   │   ├─ API 연계 데이터 자동 조회 표시
  │   │   ├─ 수기 값과 비교 보여주기
  │   │   └─ 저장 버튼 [즉시 반영]
  │   │
  │   ├─ 상태 선택 (자동 판정값 vs 수기 판정값)
  │   │   ├─ 자동 계산값 표시 (예: 달성완료 - 95% 달성)
  │   │   ├─ 다른 상태로 수정 가능
  │   │   └─ 변경 시 사유 기록
  │   │
  │   └─ 증빙 첨부 (메모/링크/파일)
  │       ├─ 파일 드래그 업로드
  │       └─ 저장 버튼
  │
  ├─ 미입력 현황
  │   ├─ 조직별 미입력 항목 수
  │   └─ 마감일 대비 입력률
  │
  ├─ 이력 조회
  │   ├─ 항목별 변경 이력
  │   ├─ 누가 언제 뭘 바꿨는지
  │   └─ 증빙 버전 이력
  │
  └─ 비교 조회 (필요시)
      ├─ 이전 기간과 비교
      └─ 초기 계획값 vs 최신값
```

### 5.3 대시보드 위젯 정의

각 Program별 기본 시각화:

1. **전체 달성 현황** (도넛 차트)
   - x축: 상태 (달성/미달성/이슈)
   - y축: 항목 수
   - 클릭: 상태별 항목 목록

2. **조직별 집계** (가로 막대)
   - x축: 조직명
   - y축: 달성률 또는 상태 분포
   - 드릴다운: 조직 내 항목 상세

3. **기간별 추이** (꺾은선)
   - x축: 기간 (1차 → 2차 → 최종)
   - y축: 달성률 또는 상태 구성비

4. **상태 분포** (스택 막대)
   - 정량 vs 정성 구분 표시
   - 색상: StatusSet에서 정의

5. **미입력/오류 현황** (테이블)
   - 항목명, 담당조직, 마감일, 입력 상태
   - 필터: 상태, 조직, 마감일 기준

6. **정량 지표 목표 대비 실적** (산점도)
   - x축: 목표값
   - y축: 실적값
   - 영역: 목표 달성선(y=x)

7. **환류 대상 항목 목록** (테이블)
   - 조건: 연속 미달성, 반복 이슈, 급격한 변동, 수기 판정 잦음
   - 컬럼: 항목명, 담당조직, 최근 상태, 환류 사유, 권장 조치
   - 목적: 단순 현황판이 아니라 회의/개선과제 도출에 바로 쓰는 목록 제공

## 9. 가정 및 제약

- **SSO 기반**: 학교 통일 SSO(LDAP/OAuth2)가 이미 운영 중
- **파일 저장**: S3 또는 온프레미스 스토리지 (구체 선택 후결정)
- **초기 데이터**: Excel로 관리자가 직접 업로드 (기존 시스템 이관 없음)
- **사용자 규모**: 관리자 5명, 부서 담당자 50-100명 (v1 기준)
- **데이터 보관**: 최소 5년 (법적 요구)
- **업무 시간**: 평일 9-18시 대량 입력 (야간/주말 부하 낮음)
- **언어**: 한국어 UI + 영문 API 문서
- **복수 지표형 처리**: 별도 타입을 만들지 않고 ItemGroup 또는 상위 분류 노드 집계로 처리
- **환류 활용**: v1 범위에서도 반복 미달성/이슈 항목 식별까지는 지원
- **정량 수집 원칙**: API 미연계 또는 API 실패 상황을 예외가 아니라 기본 시나리오로 간주하고 `manual/api/hybrid` 수집 방식을 모두 지원
- **기간 유형 원칙**: 기간 유형은 고정 enum이 아니라 관리자 정의 `typeCode`로 관리
- **finalValue 결정 원칙**: API값과 수기값이 동시에 존재할 경우 자동 덮어쓰지 않고 관리자가 최종 반영값을 명시적으로 선택
- **스냅샷 원칙**: ItemGroupSnapshot은 기준일 도달 시 초안 생성 후 관리자가 확정
- **운영 주체 원칙**: Program 생성 부서와 운영 주관 부서는 분리 가능하며, 운영 주관 부서는 기본적으로 `program_admin`
- **연도 운영 원칙**: 과제/지표/산식/담당부서 변경 가능성을 고려해 `ProgramVersion`과 `ProgramVersionItem`을 공식 운영 기준으로 사용

## 10. 위험 요소 및 완화 전략

| 위험 | 영향 | 완화 전략 |
|------|------|----------|
| SSO 서버 다운 | 로그인 불가 | 로컬 토큰 캐싱, 부분 기능(읽기) 제공 |
| API 연계 실패 | 데이터 미수집 | 수기 입력 대체, 재시도 로직, 알림 |
| 동시성 충돌 | 입력 손실 | 낙관적 잠금 + 버전 관리, 충돌 감지 |
| 대시보드 느림 | 관리자 답답함 | 집계 테이블 사전 계산, Redis 캐싱 |
| 사용자 교육 부족 | 저조한 입력률 | 매뉴얼 + 영상 튜토리얼, 운영 초기 지원 |
| 데이터 품질 | 신뢰도 저하 | 검증 규칙, 자동 이상치 감지 알림 |
