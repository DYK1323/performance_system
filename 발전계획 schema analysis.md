## 1. 전체 구조 요약

이 시스템은 전형적인 데이터베이스 중심 웹앱이라기보다, `엑셀 원본을 거의 그대로 브라우저 내 단일 페이지 앱(SPA)로 옮긴 구조`에 가깝다.

핵심 특징은 다음과 같다.

- 모든 기준 데이터가 `EXCEL_DATA`라는 단일 배열에 들어 있다.
- 각 배열 원소는 성과지표 1건을 나타내는 평면 객체이다.
- 발전부문, 발전전략, 과제, KPI, 주관부서가 별도 테이블로 분리되지 않고 각 객체 안에 문자열로 중복 저장된다.
- 사용자 입력 데이터는 별도 서버 DB 없이 `localStorage`와 일부 Google Drive 백업으로 관리된다.
- 대시보드, 목록, 입력 화면은 모두 같은 원본 배열을 재가공해 그린다.

즉, 이 시스템의 본질은 다음과 같이 요약할 수 있다.

`엑셀 행 기반 원본 데이터 + 브라우저 로컬 상태 저장 + 화면별 파생 집계`

## 2. 핵심 원본 자료 구조

### 2.1 원본 컨테이너

페이지 내부에 아래 형태의 상수가 존재한다.

```js
const EXCEL_DATA = [ ... ];
```

이 배열에는 실제로 총 `81개 지표`가 들어 있으며, 확인 결과:

- 발전부문: `7개`
- 부서: `23개`
- 지표 건수: `81개`

### 2.2 단일 지표 객체 구조

각 원소는 대체로 다음 필드를 가진다.

```js
{
  department,
  sector,
  strategy,
  task,
  kpi,
  unit,
  criteria,
  target2025,
  actual2025,
  target2026,
  target2027,
  target2028,
  target2029,
  target2030,
  evidence,
  note,
  checkCycle,
  monitorResult
}
```

### 2.3 필드 의미

- `department`: 주관부서
- `sector`: 발전부문
- `strategy`: 발전전략
- `task`: 세부 과제명
- `kpi`: 성과지표명
- `unit`: 단위
- `criteria`: 기준 및 산식
- `target2025 ~ target2030`: 연도별 목표값
- `actual2025`: 2025년 1차 원본 실적
- `evidence`: 원본 엑셀에 있던 증빙 텍스트 또는 링크
- `note`: 원본 메모
- `checkCycle`: 확인주기
- `monitorResult`: 원본 1차 모니터링 판정값

### 2.4 구조적 해석

이 객체는 정규화된 스키마라기보다 `보고서 행(row)` 중심 스키마이다. 즉,

- 분석용 계층 구조와
- 운영용 입력 데이터와
- 서술형 메모와
- 연도별 계획값

이 모두가 한 객체에 함께 담겨 있다.

장점은 단순함이다. 엑셀에서 가져온 뒤 그대로 화면에 뿌리기 쉽다.

단점은 다음과 같다.

- 같은 `strategy`, `task`, `department` 문자열이 여러 행에 중복된다.
- 특정 개념을 별도 엔터티로 관리하지 않는다.
- 변경, 검증, 참조 무결성 관리가 어렵다.
- KPI의 유형이 서로 다른데도 같은 필드 집합에 섞여 있다.

## 3. 논리적 계층 구조

화면상 의미 기준으로 보면 데이터는 다음 계층을 가진다.

`발전부문 > 발전전략 > 과제 > 성과지표 > 주관부서`

하지만 실제 구현에서는 이 계층이 별도 객체 트리로 관리되지 않는다. 대신 모든 행에 문자열로 중복 저장되어 있고, 필요할 때 `filter`, `map`, `Set`으로 재구성한다.

예를 들어:

- 발전부문 목록은 `EXCEL_DATA.map(d => d.sector)`에서 중복 제거
- 부서 목록은 `EXCEL_DATA.map(d => d.department)`에서 중복 제거
- 입력 화면의 부문별 묶음은 부서 필터 후 다시 `bySector` 객체 생성

즉, 저장 구조는 평면 배열이지만 표시 구조는 매번 동적으로 트리처럼 복원된다.

## 4. 기간 자료 구조

기간은 원본 데이터와 별도로 잘 분리되어 있다.

```js
const PERIODS = [
  {val:'2025_mid',   label:'2025년 1차',  year:2025, type:'mid',   targetKey:'target2025'},
  {val:'2025_final', label:'2025년 최종', year:2025, type:'final', targetKey:'target2025'},
  ...
]
```

### 4.1 기간 구조의 장점

- `label`, `year`, `type`, `targetKey`가 한곳에 정의되어 있다.
- 화면 코드가 특정 연도에 하드코딩되지 않고 기간 메타데이터를 참조한다.
- 중간 점검과 최종 점검의 규칙 차이를 `type`으로 제어한다.

### 4.2 기간 구조의 한계

- 실제 원본 실적은 `actual2025`만 EXCEL_DATA 내부에 존재한다.
- 이후 연도 실적은 별도 상태 객체에 저장된다.
- 결과적으로 `2025_mid`만 원본 기반, 나머지는 사용자 입력 기반이라는 비대칭 구조가 된다.

즉, 2025년 1차는 초기 정적 데이터이고, 그 외 기간은 동적 운영 데이터다.

## 5. 런타임 상태 구조

페이지는 브라우저에서 다음 상태 객체를 유지한다.

```js
let state = { actuals:{}, files:{} };
```

### 5.1 실적 상태

```js
actuals[periodVal][dept][kpi] = {
  actual,
  note,
  monitorResult
}
```

이 구조는 다음 뜻을 가진다.

- 같은 KPI라도 기간별로 다른 실적을 저장할 수 있다.
- 부서별로 데이터를 나눈다.
- KPI명을 키로 사용한다.

### 5.2 파일 상태

```js
files[periodVal][dept][kpi] = [
  { id, name, size, file?, driveLink? }
]
```

즉 증빙 파일도 같은 키 체계를 사용한다.

### 5.3 상태 키 설계 특징

식별자 전용 `id`가 없고, 사실상 다음 조합이 기본 키 역할을 한다.

`period + department + kpi`

현재 데이터셋에서는 `department + kpi` 중복이 없어 동작한다. 하지만 장기적으로 다음 위험이 있다.

- 동일 부서 안에 같은 이름의 KPI가 다시 생기면 충돌 가능
- KPI명 변경 시 기존 저장 데이터와 연결이 끊길 가능성
- 문자열 오탈자가 곧 식별자 오류가 됨

즉, 사람 친화적 키를 그대로 시스템 키로 쓴 구조다.

## 6. 저장 방식 분석

### 6.1 localStorage 저장

실제로 로컬에 저장되는 것은 `actuals` 중심이다.

```js
localStorage.setItem('bufs_kpi4', JSON.stringify({ actuals: state.actuals }));
```

이 설계의 의미:

- 실적값, 비고, 모니터링결과는 브라우저에 저장됨
- 서버 없이도 간단히 운영 가능
- 사용자 PC 또는 브라우저에 종속됨

### 6.2 파일 데이터 저장 한계

코드상 `files`는 별도 메모리 상태로 유지되지만, `localStorage`에 함께 저장되지 않는다.

따라서 구조적으로 다음 문제가 생긴다.

- 새로고침 시 파일 목록 일관성이 약할 수 있음
- 백업/복원 시 첨부 정보가 함께 복원되지 않을 수 있음
- 사용자는 파일이 "저장되었다"고 느끼지만 실제 영속성은 불완전할 수 있음

이 부분은 현재 스키마에서 가장 취약한 지점 중 하나다.

### 6.3 Google Drive 백업

Google Drive 백업은 다음 형태로 저장된다.

```js
{
  actuals: state.actuals,
  savedAt: ISODateString
}
```

여기에도 `files`는 핵심 백업 대상에 포함되지 않는다.

즉,

- 실적 데이터 백업은 가능
- 첨부파일 메타데이터는 구조적으로 부분 누락 가능
- 첨부 원본은 Drive 업로드로 따로 관리
- 하지만 앱의 내부 상태와 Drive 파일 간 완전한 일체화는 아님

## 7. 파생 자료 구조

원본 배열에서 화면에 필요한 보조 구조를 많이 만든다.

### 7.1 부문/부서 목록

```js
const SECTORS = [...new Set(EXCEL_DATA.map(d => d.sector))];
const DEPTS   = [...new Set(EXCEL_DATA.map(d => d.department))].sort();
```

### 7.2 부서별 입력 화면 묶음

입력 화면에서는:

```js
const bySector = {};
rows.forEach(d => {
  (bySector[d.sector] = bySector[d.sector] || []).push(d);
});
```

즉 부서에 속한 KPI들을 발전부문별로 다시 묶어 카드처럼 출력한다.

### 7.3 상태 색상 구조

```js
const SC = {
  '달성완료': {...},
  '달성예상': {...},
  '미달성예상': {...},
  '달성': {...},
  '미달성': {...},
  '이슈': {...},
  '': {...}
}
```

이 객체는 표현 계층용 매핑 테이블이다.

데이터 자체라기보다 `상태 -> UI 색상/배경` 사전이다.

## 8. 계산 로직과 자료형 문제

### 8.1 달성률 계산 함수

달성률 계산은 다음 식으로 처리된다.

```js
function calcAch(target, actual){
  const t = parseFloat(String(target||'').replace(/[^0-9.-]/g,''));
  const a = parseFloat(String(actual||'').replace(/[^0-9.-]/g,''));
  if(isNaN(t)||isNaN(a)||t===0) return null;
  return Math.round((a/t)*100);
}
```

### 8.2 이 함수가 잘 맞는 경우

- 단순 수치 KPI
- 백분율 KPI
- 정수 또는 소수 기반 KPI

예:

- `22`
- `75`
- `3.8`

### 8.3 이 함수가 불안정한 경우

실제 데이터에는 아래와 같은 값도 존재한다.

- `Y`, `N`
- `-`
- `예정`
- `1230(330)`
- `사회: 65.0, 공학:61.1`
- `72 이하`

이 값들은 단순 숫자 비교가 아니다.

즉 현재 스키마는 KPI 유형을 구분하지 않고 모두 문자열로 들고 있으며, 계산 함수는 그중 일부만 우연히 숫자로 환산해 처리한다.

### 8.4 의미적 문제

이 때문에 현재 구조는 다음 질문에 명확히 답하지 못한다.

- 이 KPI는 수치형인가?
- Y/N형인가?
- 범위형인가?
- 복수 지표형인가?
- 목표 방향이 "이상"인가 "이하"인가?

예를 들어 `72 이하`는 값이 낮을수록 좋은 KPI인데 현재 함수는 단순 분자/분모 비율로 이해하려고 한다. 따라서 의미상 정확한 판정 구조라고 보기 어렵다.

## 9. 모니터링 결과 구조

모니터링 결과는 기간 종류에 따라 다른 체계를 가진다.

### 9.1 중간 점검

- `달성완료`
- `달성예상`
- `미달성예상`
- `이슈`

### 9.2 최종 점검

- `달성`
- `미달성`
- `이슈`

### 9.3 자동 판정 구조

중간 점검은 대체로 달성률 기준:

- 100% 이상: `달성완료`
- 80% 이상: `달성예상`
- 그 미만: `미달성예상`

최종 점검은:

- 100% 이상: `달성`
- 그 미만: `미달성`

### 9.4 특수 규칙

`2025_mid`는 원본 엑셀의 `monitorResult`를 우선 사용한다.

즉 모니터링 결과는:

- 어떤 기간은 원본 제공값
- 어떤 기간은 계산값
- 어떤 경우는 사용자가 수정한 저장값

이 혼합 구조를 가진다.

### 9.5 자료 구조 관점의 평가

이 방식은 운영상 유연하지만, 의미적으로는 다층적이다.

- 판정 원천이 하나가 아님
- 원본값, 자동 계산값, 사용자 수정값이 공존
- 이력이나 판정 근거의 출처가 스키마에 명시되지 않음

보다 엄밀한 스키마였다면 아래 같은 필드가 분리됐을 가능성이 있다.

- `monitorResultSource`
- `monitorResultAuto`
- `monitorResultManual`
- `monitorRuleType`

현재는 모두 `monitorResult` 하나에 흡수되어 있다.

## 10. 화면별 데이터 사용 방식

### 10.1 대시보드

대시보드는 `EXCEL_DATA`를 순회하며 기간별 상태를 집계한다.

주요 산출물:

- 전체 KPI 수
- 달성/미달성/이슈/미입력 수
- 발전부문별 집계
- 부서별 지표 수
- 차수별 달성 비율 비교

즉 원본 배열을 `집계 테이블`처럼 사용한다.

### 10.2 지표 목록

목록 화면은 원본 배열을 조건 필터 후 그대로 테이블로 보여준다.

필터 기준:

- 발전부문
- 부서
- 상태
- 검색어
- 기간

즉 읽기용 상세 뷰 역할이다.

### 10.3 실적 입력

입력 화면은:

1. 부서 선택
2. 해당 부서 KPI만 필터
3. 발전부문별 그룹핑
4. 각 행에 실적 입력, 비고, 판정, 증빙을 연결

즉 동일 원본을 편집 인터페이스로 변형해 쓰는 구조다.

## 11. 구조적 장점

### 11.1 구현 단순성

- 데이터 원천이 하나라 관리가 쉽다.
- 엑셀 데이터를 바로 반영하기 쉽다.
- 서버 없이도 동작 가능하다.

### 11.2 일관성

- 대시보드, 목록, 입력 화면이 같은 원본을 참조한다.
- 기간 메타데이터가 분리돼 있어 화면 재사용성이 높다.
- 부서 모드와 관리자 모드가 같은 코드 기반에서 파생된다.

### 11.3 운영 친화성

- 작은 조직에서 빠르게 구축하기 좋다.
- 기획 변경 시 코드 수정 범위가 상대적으로 좁다.
- 엑셀 중심 업무 문화와 접점이 크다.

## 12. 구조적 한계

### 12.1 비정규화

상위 개념이 모두 문자열 중복으로 저장된다.

문제:

- 전략명 수정 시 다수 행 수정 필요
- 오탈자 발생 시 그룹핑 오류 가능
- 참조 무결성 보장 어려움

### 12.2 KPI 유형 미구분

수치형, Y/N형, 복합형, 범위형 KPI가 모두 같은 필드 형식이다.

문제:

- 판정 로직이 불안정
- 계산 결과의 의미가 KPI마다 다름
- 향후 자동화 고도화에 제약

### 12.3 문자열 기반 식별자

`department + kpi`를 사실상 키처럼 쓴다.

문제:

- 이름 변경에 취약
- 동명이표 KPI 등장 시 충돌 가능
- 사람이 읽는 이름과 시스템 키가 분리되지 않음

### 12.4 파일 구조의 불완전한 영속성

첨부파일 기능은 UI상 중요하지만 저장 구조는 상대적으로 약하다.

문제:

- 앱 상태와 Drive 파일이 완전 동기화되지 않음
- 새로고침/복원 시 일관성 우려
- 사용자 기대와 실제 저장 방식 사이 차이 발생 가능

### 12.5 2025년 1차의 예외 구조

`actual2025`, `monitorResult`는 원본 데이터에 내장되어 있고, 나머지 기간은 외부 상태에 있다.

문제:

- 기간 간 구조가 통일되지 않음
- 데이터 마이그레이션 시 특별 처리 필요
- 유지보수 시 예외 분기 증가

## 13. 사용자 친화성 평가

## 13.1 좋은 점

- 메뉴 구조가 `대시보드 / 지표 목록 / 실적 입력`으로 명확하다.
- 기간 선택 UI가 모든 주요 화면에 반복되어 현재 기준 시점이 분명하다.
- 부서별 입력 모드가 따로 있어 실제 업무 분배에 적합하다.
- 입력과 동시에 달성률이 갱신되어 피드백이 빠르다.
- 색상 배지, 카드, 차트 덕분에 상태 파악이 빠르다.
- 행별 저장과 전체 저장을 함께 제공해 실무 사용성이 높다.

## 13.2 아쉬운 점

- 테이블 폭이 넓고 정보량이 많아 모바일 친화성은 낮다.
- 중간/최종 상태 체계가 섞여 있어 일반 사용자가 상태 의미를 혼동할 수 있다.
- `criteria`가 잘려 보여 지표 해석에 필요한 맥락이 부족하다.
- `evidence`와 첨부파일 팝업이 분리되어 증빙 위치에 대한 정신 모델이 일관되지 않다.
- 복합 KPI나 Y/N KPI는 달성률 표시가 직관적이지 않다.
- 권한 모델이 부서명 분기에 일부 의존해 사용자 역할 체계가 명확하지 않다.

## 13.3 사용자 경험 총평

이 시스템은 `처음부터 완전한 정보시스템으로 설계된 제품`이라기보다, `엑셀 기반 행정 업무를 웹 화면으로 효율화한 실무형 도구`에 가깝다.

따라서 사용자 친화성은 다음처럼 평가할 수 있다.

- 단기 실무 운영 관점: 비교적 친절함
- 데이터 모델의 엄밀성 관점: 다소 취약함
- 확장성과 일관성 관점: 개선 필요

한마디로 정리하면:

`UI는 실무자에게 꽤 친절하지만, 그 친절함을 떠받치는 자료 구조는 아직 엑셀 중심 설계의 한계를 강하게 안고 있다.`

## 14. 결론

현재 페이지의 자료 구조는 다음 성격을 가진다.

- 강점: 빠른 구축, 쉬운 이해, 엑셀 친화성, 화면 일관성
- 약점: 비정규화, 문자열 키 의존, KPI 유형 혼합, 파일 영속성 약함, 기간 구조 비대칭

따라서 이 시스템은 `운영 프로토타입 또는 실무형 경량 관리도구`로는 매우 유용하지만, 장기적으로 고도화하려면 최소한 아래 4가지는 재설계 대상이다.

- KPI 고유 ID 도입
- KPI 유형 스키마 분리
- 실적/판정/증빙 구조 분리
- 파일 메타데이터와 백업 구조 일원화

필요하다면 다음 단계로 이어서 정리할 수 있다.

- 개선안 중심의 `to-be schema` 설계
- ERD 형태 재구성
- 사용자 친화성 개선 제안서 작성
